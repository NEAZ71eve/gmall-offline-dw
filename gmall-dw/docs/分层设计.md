# 数仓分层设计

## 一、分层架构概述

| 层级 | 名称 | 职责 | 存储格式 | 表命名规范 | 生命周期 |
|------|------|------|---------|-----------|---------|
| ODS | 原始数据层 | 直接同步业务库与日志，不清洗、不加工 | Text | ods_* | 3个月 |
| DIM | 公共维度层 | 用户、商品、店铺、时间等维度表 | ORC | dim_* | 永久 |
| DWD | 明细数据层 | 清洗、脱敏、去重、统一规范 | ORC | dwd_* | 1个月 |
| DWS | 汇总数据层 | 按主题轻度聚合（用户、订单、商品等） | ORC | dws_* | 永久 |
| ADS | 应用数据层 | 指标报表、大屏、接口输出 | ORC | ads_* | 永久 |

## 二、各层详细设计

### 2.1 ODS层（原始数据层）

**职责**：
- 直接从业务数据库同步数据
- 保持原始数据格式，不做任何加工
- 用于数据追溯和审计

**表设计原则**：
- 与源表结构保持一致
- 使用Text格式存储
- 按天分区（dt）

**典型表**：
- ods_user_info - 用户信息表
- ods_order_info - 订单信息表
- ods_order_detail - 订单明细表
- ods_sku_info - 商品SKU表
- ods_activity_info - 活动信息表

**生命周期策略**：
- 保留周期：3个月
- 清理策略：每月1日清理3个月前的数据
- 存储策略：冷数据转储到归档存储

### 2.2 DIM层（公共维度层）

**职责**：
- 存储维度数据
- 处理缓慢变化维（SCD1/SCD2/SCD3）
- 提供统一的维度编码

**表设计原则**：
- 使用ORC压缩格式
- 支持多种SCD类型
- 存储历史快照

**典型表**：
- dim_user_scd1 - 用户维度表（SCD1）
- dim_user_scd2 - 用户维度表（SCD2拉链表）
- dim_user_scd3 - 用户维度表（SCD3）
- dim_sku - 商品SKU维度表
- dim_spu - 商品SPU维度表
- dim_brand - 品牌维度表
- dim_category1/2/3 - 分类维度表（雪花模型）
- dim_time - 日期维度表
- dim_province - 地区维度表

**生命周期策略**：
- 保留周期：永久
- 存储策略：热数据存储

### 2.3 DWD层（明细数据层）

**职责**：
- 数据清洗（去重、过滤脏数据）
- 数据脱敏（手机号、邮箱等）
- 格式统一（时间格式、编码统一）
- 关联维度键

**表设计原则**：
- 使用ORC压缩格式
- 事务型事实表
- 保留明细数据

**典型表**：
- dwd_order_detail - 订单明细事实表
- dwd_order_info - 订单事实表
- dwd_payment - 支付事实表
- dwd_refund - 退款事实表

**生命周期策略**：
- 保留周期：1个月
- 清理策略：每月1日清理1个月前的数据
- 存储策略：热数据存储（最近7天），温数据存储（7-30天）

### 2.4 DWS层（汇总数据层）

**职责**：
- 按主题轻度聚合
- 按维度汇总指标
- 提供跨主题关联

**表设计原则**：
- 使用ORC压缩格式
- 按天/周/月聚合
- 多粒度汇总

**典型表**：
- dws_user_day - 用户每日汇总
- dws_order_day - 订单每日汇总
- dws_sku_day - 商品每日汇总
- dws_province_day - 地区每日汇总
- dws_user_week - 用户每周汇总
- dws_user_month - 用户每月汇总

**生命周期策略**：
- 保留周期：永久
- 存储策略：热数据存储

### 2.5 ADS层（应用数据层）

**职责**：
- 计算最终指标
- 为报表和大屏提供数据
- 提供API接口数据

**表设计原则**：
- 使用ORC压缩格式
- 面向业务指标
- 多时间粒度（1日/7日/30日）

**典型表**：
- ads_gmv_day - GMV日报表
- ads_user_retention - 用户留存表
- ads_conversion_rate - 转化率表
- ads_sku_sales_rank - 商品销售排行
- ads_province_stats - 地区销售统计

**生命周期策略**：
- 保留周期：永久
- 存储策略：热数据存储

## 三、分区策略

| 层级 | 分区方式 | 说明 |
|------|---------|------|
| ODS | dt (日期) | 按天分区 |
| DIM | dt (日期) | 拉链表按天更新 |
| DWD | dt (日期) | 按天分区 |
| DWS | dt (日期) | 按天分区 |

## 四、数据生命周期管理

### 4.1 生命周期策略表

| 层级 | 保留周期 | 热数据期限 | 温数据期限 | 冷数据期限 | 清理策略 |
|------|---------|-----------|-----------|-----------|---------|
| ODS | 3个月 | 7天 | 30天 | 90天 | 每月1日清理 |
| DIM | 永久 | 全部 | - | - | 不清理 |
| DWD | 1个月 | 7天 | 30天 | - | 每月1日清理 |
| DWS | 永久 | 全部 | - | - | 不清理 |
| ADS | 永久 | 全部 | - | - | 不清理 |

### 4.2 冷热数据分离策略

**热数据**（最近7天）：
- 存储位置：HDFS SSD存储
- 访问频率：高
- 备份策略：多副本

**温数据**（7-30天）：
- 存储位置：HDFS HDD存储
- 访问频率：中等
- 备份策略：单副本

**冷数据**（30天以上）：
- 存储位置：归档存储（如HDFS Archive或外部存储）
- 访问频率：低
- 备份策略：归档

### 4.3 生命周期管理脚本

```sql
-- ODS层生命周期管理
ALTER TABLE ods_user_info SET TBLPROPERTIES (
    'retention.days' = '90',
    'retention.policy' = 'delete'
);

-- DWD层生命周期管理
ALTER TABLE dwd_order_detail SET TBLPROPERTIES (
    'retention.days' = '30',
    'retention.policy' = 'delete'
);
```

## 六、开发规范

### 6.1 SQL编码规范

1. **关键字大写**：SELECT, FROM, WHERE, JOIN等关键字使用大写
2. **缩进规范**：每个子句换行，使用4空格缩进
3. **命名规范**：表名、字段名使用小写，下划线分隔
4. **注释规范**：复杂SQL必须添加注释说明业务逻辑

### 6.2 ETL开发规范

1. **数据校验**：每个ETL任务必须包含数据质量校验
2. **日志记录**：关键节点必须记录日志
3. **异常处理**：必须处理异常情况，保证任务稳定性
4. **幂等性**：ETL任务必须保证幂等，可重复执行

### 6.3 代码审查规范

1. **审查内容**：SQL逻辑正确性、性能优化、命名规范、注释完整性
2. **审查流程**：开发完成→自测→提交审查→合并
